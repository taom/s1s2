# S1S2 Support Playbook

## Purpose

This document defines how founder-led support should work through beta, launch, and early growth.

## Support Voice

- precise, warm, and calm
- never defensive
- never diagnostic
- always honest about known issues and current limits
- always protect the player's dignity and trust

## Channels

- support email
- in-app support link
- store review response workflow
- limited social DM redirection back to support email for account-specific issues

## Severity Model

| Severity | Definition | Target Response |
|---|---|---|
| Sev 0 | active data loss, privacy issue, broken deletion/export, widespread launch blocker | same day |
| Sev 1 | scan blocked, app crash loop, purchase broken, progress locked | within 1 business day |
| Sev 2 | incorrect display, import confusion, reminder problems, minor entitlement mismatch | within 2 business days |
| Sev 3 | feature request, cosmetic issue, content feedback | within 3 business days |

## Issue Classes

- onboarding and activation
- camera scan quality
- manual logging and import issues
- journey or creature progression issues
- audio and playback issues
- purchase and entitlement issues
- privacy, export, or deletion issues
- trust or tone complaints

## Triage Workflow

1. classify severity
2. classify issue type
3. confirm whether the issue is user-specific or systemic
4. check whether the issue affects trust, retention, or revenue
5. route to bug list, known issue list, or direct support resolution

## Required Support Metadata

For every ticket, capture:

- platform and OS
- app version
- account email or identifier
- issue type
- severity
- reproduction steps if available
- screenshot or screen recording if offered
- whether the issue affected trust, data accuracy, or payment

## Health-Adjacent Escalation Rules

- if a user asks whether a result means something is medically wrong, do not interpret the data
- advise that S1S2 is not a diagnostic medical device
- direct the user to a healthcare professional for medical concerns
- if the issue suggests self-harm or crisis, prioritize a safety-oriented response and direct toward local emergency or crisis resources

## Store Review Response Rules

- thank the reviewer briefly
- acknowledge the issue clearly
- avoid arguing about the complaint in public
- point to the support channel for account-specific help
- fix product-side wording if the same complaint repeats

## Macro Outlines

### Scan Quality Reply

- acknowledge the frustration
- explain that low-confidence scans can happen due to lighting, finger temperature, movement, or camera coverage
- offer the best next steps
- mention manual logging or imported HR as available fallback paths if applicable

### Purchase or Restore Reply

- acknowledge the issue
- ask for platform and purchase context
- confirm restore steps
- explain the expected entitlement behavior
- escalate quickly if a paid user lost access

### Privacy or Export Reply

- explain what data is stored
- explain what can be exported or deleted
- provide the next concrete step
- escalate if the user reports missing deletion or export functionality

### Known Bug Reply

- acknowledge the bug directly
- avoid overpromising a date
- explain the workaround if one exists
- confirm the issue has been logged

## Beta Support Rules

- respond within 24 hours during closed beta and launch month
- tag every beta issue for activation, trust, quality, or delight impact
- prefer fixing first-hour experience issues over long-tail polish items
